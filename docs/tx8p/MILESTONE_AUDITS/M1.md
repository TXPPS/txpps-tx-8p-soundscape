# Milestone 1 — v4 Refinement Audit

Status: **Awaiting visual approval** — audio not yet started.

## 1. Root cause of the previous performance-layout mismatch

v3 rendered the performance area as a **tall vertical `perf-panel` card
on the left of the control surface**, sitting *next to* the editor and
*above* a separately framed keyboard. Pitch and mod were oversized
levers stacked in a chassis panel that consumed valuable vertical space,
Hold and Octave were full-width bars, and a large MODE readout duplicated
what the VOICE tab already owned. Keyboard and controls were two
independent chassis panels — visually detached, ergonomically wrong for
a live playing surface.

v4 restructures the entire lower half around the TX27 layout logic:
**one shared recessed chassis bed** that contains Pitch/Mod strips, a
compact octave utility column, and the keyboard, in that horizontal order.

## 4. Pitch control redesign

- Narrow charcoal recessed track, 22 px visible width, 44 px hit target.
- Bipolar with a subtle centre reference tick.
- 3 px cream indicator with soft glow, follows drag position.
- Spring-to-centre wired on `pointerup` via `onRelease`.
- Small engraved `PITCH` label at the bottom (or leading edge in horizontal mode).

## 5. Mod control redesign

- Identical form factor to Pitch, unipolar (0..1), amber indicator.
- No spring; retains position on release.
- Sits directly beside Pitch in the left rail.

## 6. Octave control redesign

- Compact `OCT +` / `OCT −` cream program buttons stacked vertically.
- Small amber LCD (`+0`, `+1`, `-1`, …) using the same smoked-glass language
  as the main preset LCD.
- Replaces the earlier full-width `−` LCD `+` bar.

## 7. Hold redesign

- Single compact amber program button labelled `HOLD`.
- Slots into the octave utility column beneath the octave LCD (desktop) or
  at the end of the utility row (phone portrait).
- Active state clearly indicated via the button's LED dot + amber press state.
- No longer spans the full width of a panel.

## 8. Voice-mode relocation

- The dedicated MODE readout has been **removed** from the performance
  deck. Detailed voice-mode editing continues to live on the VOICE tab.
- Mode can be surfaced on the main LCD when needed; the performance deck
  no longer sacrifices any horizontal space for it.

## 9. Keyboard proportion changes

- White keys: warm ivory (`--key-white`) with subtle vertical shadow,
  1 px charcoal separator, minimal `0 0 2px 2px` corner radius.
- Black keys: deep charcoal, 60 % white-key width, **62 %** white-key
  length, thin top highlight, restrained shadow beneath.
- Keyboard now fills the remaining horizontal space after the perf rail
  (4 octaves on desktop, 2 octaves on phone portrait).
- No outer floating card — the keyboard sits inside the shared deck bed
  with no gap between rail and keys.

## 10. Desktop result (1280 × 800)

- Header, tab row, focused editor, and full performance deck are all
  visible without horizontal scroll.
- Deck docks to the bottom of the viewport (`mt-auto`).
- Keyboard spans ~1100 px and shows 4 full octaves (28 white keys).
- Screenshot: `/tmp/browser/m1v4/screenshots/desktop.png`.

## 11. Phone portrait result (390 × 844)

- Two-row identity bar, tabs, editor, then the deck.
- Deck: compact row of `OCT − · +0 · OCT + · HOLD`, then horizontal
  `PITCH` and `MOD` strips, then a 2-octave keyboard anchored at the bottom.
- The keyboard is reachable in a single scroll and dominates the lower half.
- Screenshot: `/tmp/browser/m1v4/screenshots/phone_p.png`.

## 12. Phone landscape result (844 × 390)

- Deck uses the same row layout as desktop (left rail + keyboard), which
  matches the TX27 landscape ergonomics.
- At 390 px viewport height the header + tabs + editor + deck exceed the
  fold, so the deck is reached by one short scroll. A dedicated
  editor-collapse-on-landscape drawer is deferred to a later polish pass
  (documented here so it is not lost) and does not change the M1 v4
  hardware/layout intent.
- Screenshot: `/tmp/browser/m1v4/screenshots/phone_l.png`.

## 13. Accessibility checks

- `PitchModStrip` exposes `role="slider"` with `aria-valuemin/max/now`
  and is keyboard-focusable.
- All hit targets ≥ 44 × 44 px (strip visible width is 22 px, box is
  44 px; program buttons remain 44 × 26 with a 44 px min-width).
- `touch-none` prevents page scroll while dragging a strip.
- Global `pointermove` / `pointerup` / `pointercancel` listeners guarantee
  the drag ends cleanly on system gestures, cancelled captures, or blur.
- Reduced-motion path preserved from earlier milestones.

## 14. Verification

- `bunx tsgo --noEmit`: clean.
- Playwright: zero `pageerror`, zero console warnings/errors on
  desktop 1280 × 800, phone portrait 390 × 844, and phone landscape
  844 × 390.
